# Migrating to v4.0

This guide covers all breaking changes in EUDIPLO v4.0 and the steps required to upgrade from v3.x.

## Overview of Changes

v4.0 introduces a **unified Key Chain model** that replaces the separate Key and Certificate entities. This is the most significant structural change and affects the API, configuration import format, and database schema.

Other changes include:

- **API field renames** across credential configs, trust lists, status lists, and presentations
- **Configuration import format** changed from separate key/cert folders to unified key-chains
- **New key chain features**: rotation policies, usage types, and export capability

## Pre-Upgrade Checklist

- [ ] Back up your database
- [ ] Back up your configuration import files (if using `CONFIG_IMPORT`)
- [ ] Note your current environment variables
- [ ] Review the sections below for all changes affecting your setup

## Database Migration

### What happens automatically

v4.0 includes a migration that runs on startup. The migration system will:

1. Detect whether your database was created by a previous version (via `synchronize: true`)
2. Apply incremental schema changes (adding columns, renaming fields)
3. **Migrate your existing keys and certificates** into the new `key_chain` table
4. Drop the legacy `key_entity`, `cert_entity`, `key_usage_entity`, and `cert_usage_entity` tables

!!! success "No manual SQL required"
All database changes are handled automatically. Just start the new version and the migrations will run.

### Environment variable changes

No new environment variables are required for v4.0. The `DB_MIGRATIONS_RUN=true` default (introduced in v3.0) handles everything.

## API Breaking Changes

### Unified Key Chain model

The separate Key and Certificate APIs have been replaced by a single **Key Chain** API.

| Old Endpoint         | New Endpoint                                            |
| -------------------- | ------------------------------------------------------- |
| `POST /key/import`   | `POST /key-chain/import`                                |
| `POST /key/generate` | `POST /key-chain/generate`                              |
| `GET /key`           | `GET /key-chain`                                        |
| `GET /key/:id`       | `GET /key-chain/:id`                                    |
| `DELETE /key/:id`    | `DELETE /key-chain/:id`                                 |
| `POST /cert/import`  | _(removed — certificates are part of key chain import)_ |
| `GET /cert`          | _(removed)_                                             |
| `GET /cert/:id`      | _(removed)_                                             |

### Field renames

All references to `certId` have been renamed to `keyChainId` across the API:

#### Credential Configuration

| Old Field | New Field    |
| --------- | ------------ |
| `certId`  | `keyChainId` |

#### Trust List

| Old Field          | New Field              |
| ------------------ | ---------------------- |
| `certId`           | `keyChainId`           |
| `issuerCertId`     | `issuerKeyChainId`     |
| `revocationCertId` | `revocationKeyChainId` |

#### Status List

| Old Field | New Field    |
| --------- | ------------ |
| `certId`  | `keyChainId` |

Applies to all status list endpoints: create, update, import, and response DTOs.

#### Presentation Configuration

| Old Field      | New Field          |
| -------------- | ------------------ |
| `accessCertId` | `accessKeyChainId` |

#### Issuance Configuration

No field renames. `signingKeyId` remains unchanged.

### Key Chain import format

The import payload has changed from separate key + certificate imports to a unified format:

**Old format** (two separate calls):

```json
// POST /key/import
{ "kid": "my-key", "key": { "kty": "EC", ... } }

// POST /cert/import
{ "keyId": "my-key", "crt": ["-----BEGIN CERTIFICATE-----..."] }
```

**New format** (single call):

```json
// POST /key-chain/import
{
    "id": "my-key-chain",
    "description": "My attestation key",
    "usageType": "attestation",
    "key": { "kty": "EC", "crv": "P-256", "x": "...", "y": "...", "d": "..." },
    "crt": ["-----BEGIN CERTIFICATE-----..."],
    "kmsProvider": "internal",
    "rotationPolicy": {
        "enabled": false
    }
}
```

### Usage types

Certificate "type" flags have been replaced by explicit `usageType` values:

| Old (cert flags)      | New (`usageType`) |
| --------------------- | ----------------- |
| `isSigningCert: true` | `"attestation"`   |
| `isAccessCert: true`  | `"access"`        |
| _(status list cert)_  | `"statusList"`    |
| _(trust list cert)_   | `"trustList"`     |

## Configuration Import Format

If you use `CONFIG_IMPORT=true` with a config folder, the directory structure has changed:

### Old structure

```
config/
  my-tenant/
    keys/
      signing-key.json
      access-key.json
    certs/
      signing-cert.json
      access-cert.json
    issuance/
      ...
```

### New structure

```
config/
  my-tenant/
    key-chains/
      attestation.json
      access.json
      status-list.json
      trust-list.json
    issuance/
      ...
```

Each key chain JSON file contains both the key and certificate:

```json
{
    "id": "unique-key-chain-id",
    "description": "Attestation key for credential signing",
    "usageType": "attestation",
    "key": {
        "kty": "EC",
        "crv": "P-256",
        "x": "...",
        "y": "...",
        "d": "..."
    },
    "crt": ["-----BEGIN CERTIFICATE-----\n...\n-----END CERTIFICATE-----"],
    "kmsProvider": "internal",
    "rotationPolicy": {
        "enabled": false
    }
}
```

!!! tip "Export existing key chains"
If you created key chains via the web client and want to convert them to config import format, use the new export endpoint: `GET /key-chain/:id/export`

## Step-by-Step Upgrade

1. **Stop the running instance**

2. **Back up your database**

    ```bash
    # For SQLite
    cp /path/to/service.db /path/to/service.db.backup

    # For PostgreSQL
    pg_dump -h localhost -U eudiplo eudiplo > backup.sql
    ```

3. **Update environment variables**

    ```bash
    # Remove or set to false
    DB_SYNCHRONIZE=false

    # Ensure migrations run (this is the default)
    DB_MIGRATIONS_RUN=true
    ```

4. **Update configuration import files** (if using `CONFIG_IMPORT`)
    - Rename `keys/` and `certs/` folders to `key-chains/`
    - Merge key + cert files into unified key chain JSONs (see format above)

5. **Update API integrations**
    - Replace `certId` with `keyChainId` in all API calls
    - Replace `/key/` and `/cert/` endpoints with `/key-chain/`
    - Update import payloads to the new unified format

6. **Deploy the new version**

    ```bash
    # Docker
    docker pull ghcr.io/openwallet-foundation-labs/eudiplo:4.0.0
    docker compose up -d
    ```

7. **Verify startup**
    - Check logs for migration output: `[Migration] MigrateKeysToKeyChain: Found N keys to migrate.`
    - Verify the service responds on its health endpoint
    - Test your issuance and presentation flows

## Rollback

If you need to roll back:

1. Stop the v4.0 instance
2. Restore your database backup
3. Re-deploy the previous version

!!! warning
The v4.0 migration drops legacy tables. Without a database backup, rollback is not possible.
